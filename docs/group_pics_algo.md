# 群照人物算法演进记录

本文记录群照相关的人脸识别与人物归属逻辑演进，主要用于复盘。

这里的“版本”指主导思路的变化，不追求把每一次局部调参都单独拆成一版。

## v1：参考脸驱动的候选照片匹配

### 核心目标

给定目标人物的参考脸，从候选照片里找出“同时包含这些目标人物”的照片。

这个版本的重心是“找照片”，不是“给全库每一张脸做长期归属”。

### 典型输入

- 人物 A 的参考图目录
- 人物 B 的参考图目录
- 一批候选照片

参考图要求比较严格：

- 每张参考图必须只有一张可用人脸
- 参考图数量可以是多张
- 参考图的作用是定义目标人物长什么样

### 主流程

1. 先读取参考图，为每个目标人物建立参考表示。
2. 再对候选照片逐张做人脸检测。
3. 对候选照片中的每一张候选脸，分别计算它和目标人物参考表示之间的距离。
4. 如果一张候选照片里，A 和 B 都能各自命中，而且命中的是两张不同的脸，这张照片就算命中。
5. 最后再把命中的照片分到 `only-two` 或 `group`。

### 参考表示

v1 的大方向始终是“给参考脸，再去匹配”。

这一代内部经历过两种典型做法，但都属于同一条主线：

- 早期做法：直接保留多张参考脸 embedding，候选脸对该人物的距离取“到参考集合的最小距离”。
- 后期增强：把多张参考脸组织成参考模板，加入轻量清洗、样本质量分数、模板中心和 `top-k mean` 之类的稳健分数。

无论具体实现是哪一版，v1 的本质都没有变：

- 参考图是算法起点
- 候选图是被检索对象
- 判断结果是“这张照片是否包含目标人物”

### 照片命中规则

v1 的命中语义是照片级的，不是 observation 级或 assignment 级的。

它关心的是：

- 这张候选照片里是否出现了目标人物 A
- 这张候选照片里是否出现了目标人物 B
- A 和 B 是否对应到两张不同的人脸

满足这些条件后，这张照片才会被认为是目标合照。

### `only-two` / `group` 规则

v1 不只是判断“命中没命中”，还会继续判断照片属于双人图还是多人图。

主思路是：

- 如果只检测到两张脸，通常归到 `only-two`
- 如果检测到的人脸更多，就继续看除主匹配对之外，是否还有足够显著的额外人脸
- 如果有明显第三人或更多人，就归到 `group`
- 如果额外人脸都非常小，仍可能保留在 `only-two`

所以 v1 的输出本质上是“照片检索 + 照片分桶”。

### v1 的优点

- 用户心智简单：先给参考脸，再找相似照片
- 对没有完整人物库、只想快速找合照的场景很直接
- 容易做人工调参和局部复核

### v1 的局限

- 依赖参考图质量，参考图一旦带偏，结果就会一起偏
- 更像一次性的检索流程，不适合沉淀人物长期身份记录
- 很难把“纠错结果”自然沉淀成后续全库能力
- 更适合回答“这张图像不像目标人物”，不擅长维护“这个 observation 长期归属于谁”

## v2：人物原型驱动的全库人脸归属

### 核心目标

把照片库中的每一张人脸 observation，持续归属到某个人物记录上，并允许后续修正、排除、复核和导出。

这个版本的重心从“找照片”切换成了“建人物库、做归属、再基于归属去用照片”。

### 核心对象

v2 的几个关键对象是：

- 照片资产
- 人脸 observation
- observation 的 embedding
- 人物记录
- 人物原型
- observation 到人物的 assignment

这意味着 v2 的主语不再是“参考图目录”，而是“人物记录”和“人脸归属关系”。

### 主流程

1. 新照片入库。
2. 对照片做人脸检测，生成多个 observation。
3. 对每个 observation 裁脸并提取 embedding。
4. 从当前人物库中读取每个人的 active 原型。
5. 用 ANN 先召回一批候选人物。
6. 再对 observation 和候选人物原型计算精确距离。
7. 选距离最近的人物作为第一候选。
8. 根据阈值决定：
   - 直接自动归属
   - 进入低置信度待复核
   - 进入新人物候选

### 人物原型

v2 不再要求用户先给参考图目录。

人物的表示来自当前已经归属到这个人物的 active observation：

- 先收集这个人物当前所有 active 样本
- 再从这些样本中构建人物原型
- 当前主表示可以理解为一个人物 centroid

所以 v2 的人物表示是“从已有归属数据反推出来的”，而不是“从用户手工提供的参考图直接定义出来的”。

### 归属判定方式

v2 当前的归属判定是 observation 级、独立式的。

也就是说：

- 一张照片里如果有 10 张脸，这 10 张脸会分别独立判断
- 每张脸都单独去和人物原型比较
- 当前实现主要看第一名候选的绝对距离
- 如果第一名距离落进自动归属区间，就直接写入 assignment

这一代的一个重要特征是：

- 它默认在回答“这张脸是谁”
- 而不是直接回答“这张照片是不是某两个人的合照”

### 人工纠错如何影响后续

v2 的一大变化，是纠错不再只是这一次检索的临时操作，而会回写到底层人物数据。

例如：

- 手动排除错误归属，会让该 observation 不再被归到这个人物
- 排除后，这个人物的 active 样本集合会变化
- active 样本集合变化后，人物原型也会随之变化
- 人物原型变化后，后续自动归属结果也会变化

所以 v2 天然支持“纠错后逐步收敛”。

### 导出模板在 v2 中的位置

v2 里模板导出已经不是底层识别入口，而是归属结果的下游消费。

也就是说：

- 先有人脸 observation 的归属结果
- 再根据模板要求的人物集合，筛选哪些照片命中

因此，导出命中的质量强依赖前面的 assignment 质量。

### v2 的优点

- 适合长期维护人物库和归属历史
- 支持人工纠错、批量排除、后续复核
- 支持从“人”出发查看样本、修正原型，再反向影响未来结果
- 更适合做人物详情页、归属记录、导出模板、回溯审计这些能力

### v2 当前暴露出的主要问题

复盘下来，v2 的问题不在于“没有人物库”，而在于自动归属策略仍然偏激进。

目前比较明显的风险点有：

- 自动归属更看重第一名的绝对距离，缺少对第一名和第二名间隔的约束
- 一张照片中的多张脸是彼此独立判断的，缺少照片级联合约束
- 小脸、糊脸、远景脸也可能直接进入自动归属
- 如果错误归属积累进人物原型，后续还会继续放大误报
- 如果人物原型直接由全部自动归属样本计算，自动归属里的误差会再次写回原型，形成自我强化
- 如果原型维护主要依赖“先自动归属，再靠排除做减法”这条路径，用户纠错成本会迅速上升

所以 v2 的核心问题更像是“归属决策和原型维护还不够保守”，而不是方向本身错误。

### v2 复盘后的补充判断

从这轮复盘看，人物原型的构建原则需要单独拎出来看。

更合理的方向应该是：

- 人物原型不应直接由全部自动归属样本决定
- 人物原型应优先由高信任样本来构建
- 高信任样本更适合来自人工确认样本、用户认为高质量的样本，或极高置信且长期稳定的自动归属样本

换句话说，原型更新更适合“确认做加法”，而不是“排除做减法”。

原因是：

- 加法策略更符合用户心智，用户只需要持续确认“哪些样本真的属于这个人”
- 减法策略要求用户不断清理历史误归属，工作量会随着图库规模增长而快速放大
- 一旦误归属先进入原型，再靠后续排除去修，系统会天然慢一拍，并且容易在中间阶段持续污染后续结果

因此，从复盘角度，v2 后续演进的重点不只是把自动归属判定调得更保守，还包括把人物原型的数据来源从“全量 active 样本”逐步收敛到“高信任确认样本池”。

## v3：高信任样本池驱动的人物重建

### 核心目标

v3 的方向，不再是基于现有归属结果继续调阈值，而是把人物层看成一套需要重新长出来的结构。

这个版本的重心是：

- 先把 observation 自身质量和 identity 判定拆开
- 再从高质量 observation 里长出更干净的人物
- 让人物原型只由高信任正样本池构建

所以 v3 的核心问题不再只是“这张脸最像谁”，而是：

- 哪些 observation 适合参与人物表示
- 哪些 observation 只是暂时归属
- 哪些 observation 足够可靠，可以定义一个人物

### 质量门控

v3 的一个关键变化，是把 `quality_score` 从辅助信息提升为主流程约束。

这一代更明确地区分了两类信号：

- observation 质量：这张脸本身清不清楚、够不够大、适不适合代表人物
- identity 置信：这张脸像不像某个人

也就是说：

- `quality_score` 回答“这张脸好不好”
- `confidence` 回答“这张脸像不像某个人”

这两个概念拆开以后，低质量 observation 就算勉强命中某个人，也不应该自动进入人物原型。

从当前演进方向看，v3 的首版质量门更适合优先依赖：

- 人脸面积
- 清晰度

姿态分仍然是后续可补的方向，但不再作为首版前提。

### 新人物发现方式

v2 里的“新人物聚类”更像 review 页里的辅助视图，而不是人物库主流程的一部分。

v3 在这件事上的思路更激进一些：

- 未可靠归属的高质量 observation 不应只是堆在待审核里
- 它们应该先在未归属池里形成稳定 cluster
- 当 cluster 足够稳定时，系统可以先长出匿名人物
- 用户后续主要做的是命名、确认、忽略、合并，而不是从零开始手工建人物

所以 v3 的人物增长路径，更像是“先自动发现，再人工校正”，而不是“先人工建人，再允许系统归属”。

### 人物原型

v3 对人物原型的态度，比 v2 更保守。

这一代的关键变化是：

- 人物原型不再默认由全部 active assignment 决定
- 人物原型更适合由 trusted positive pool 决定
- trusted positive pool 主要来自高质量人工确认样本
- 也可以吸收少量极高置信、长期稳定的自动样本

这意味着 v3 更强调把“当前归属”和“可用于定义这个人长什么样的样本”分开。

换句话说：

- 普通 auto assignment 可以解决“这张脸大概率是谁”
- 但它不应自动拥有“修改人物原型”的资格

### 归属判定方式

v3 仍然是人物库驱动的 observation 级归属，但判定会比 v2 更保守。

更明显的变化包括：

- 自动归属不再只盯着第一名的绝对距离
- 会更重视第一名和第二名之间的间隔
- 会更重视 observation 本身质量
- 会把照片级冲突和歧义纳入判断

因此，v3 里的 observation 更容易分成三类：

- 可以稳定归到已有人物
- 暂时不能稳归旧人物，但可以和其他 observation 一起长成新人物
- 既不能稳定归旧人物，也不足以长成新人物，需要等待或复核

### v3 的优点

- 更符合大图库场景下“先保证样本干净，再逐步扩人”的需求
- 能把新人物发现正式纳入主流程，而不是长期滞留在 review 辅助层
- 更能控制误归属对人物原型的污染
- 更符合“确认做加法”的用户工作流

### v3 仍然要面对的代价

- 系统前期会比 v2 更保守，自动召回率可能先下降
- 需要单独维护高信任样本池，而不是只维护 assignment
- bootstrap 的设计会直接影响人物库初始质量

但从复盘角度看，这些代价是值得的。

因为真正需要避免的，不是“初期少自动一些”，而是“错误样本先污染人物原型，再把误差越滚越大”。

## 当前结论

如果把这几代思路串起来看：

- v1 解决的是“给参考脸，找照片”
- v2 解决的是“建人物库，做全库归属”
- v3 要解决的则是“如何在大图库里用更干净的样本机制重建人物，并避免错误原型自我强化”

所以 v3 的关键演进，不只是把自动归属调得更严，而是把人物层的生长基础从“全量自动归属样本”切换成“高质量、高信任样本池”。
