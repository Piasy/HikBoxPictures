# HikBox Pictures 设计文档

## 目标

构建一个本地 macOS CLI 工具，递归扫描照片目录，找出同时包含指定两个人的照片，并把命中的文件复制到新的输出目录中。命中结果必须分成两类：只包含这两个人的照片，以及除了这两个人之外还有其他人的照片。

## 范围

第一版是一个运行在 macOS 上的本地 Python CLI。它处理任意目录中的文件，而不是 Apple Photos 资料库。工具会递归扫描图片文件，使用用户提供的两张参考照片做人脸匹配，并把命中的文件复制到按拍摄月份组织的输出目录中。

范围内：
- 递归扫描 `HEIC`、`JPG`、`JPEG` 和 `PNG`
- 两张参考照片，每个人一张
- 完全本地的人脸匹配
- 输出分为 `only-two` 和 `group`
- 输出子目录按 `YYYY-MM` 命名
- 对命中的 `HEIC` 复制其对应的 Live Photo `MOV` 文件
- 复制后保留源文件时间元数据

范围外：
- 视频内容分析
- Apple Photos / PhotoKit 集成
- 第一版的跨平台打包
- 持久化人脸数据库或具名身份管理

## 已选方案

使用基于 `face_recognition` 的 Python CLI。

选择原因：
- 它直接支持从参考照片提取人脸编码，并与候选图片中的人脸进行比对。
- 它和“每个人提供一张参考照片”的用户流程天然匹配。
- 它完全本地运行，不需要 Photos 资料库集成，也不需要云端 API。

考虑过的替代方案：
- Apple `Vision` / `PhotoKit`：macOS 原生集成更强，但对于“扫描任意目录的 CLI”不够贴合，而且对“两张参考照做身份匹配”也没有同等直接。
- 云端识别 API：因为需求是本地处理和私密照片处理，所以不采用。

## 用户流程

CLI 接受 4 个必填参数：

```bash
hikbox-pictures \
  --input /path/to/photo-library \
  --ref-a /path/to/person-a.jpg \
  --ref-b /path/to/person-b.jpg \
  --output /path/to/output
```

行为流程：
1. 读取两张参考照片。
2. 校验每张参考照片都只能检测到且必须检测到一张人脸。
3. 递归扫描输入目录中的受支持图片文件。
4. 对每张候选图片执行人脸检测和匹配。
5. 如果目标两人同时出现，则将图片归类到两个结果桶之一。
6. 从拍摄时间元数据确定年月；如果拿不到，再逐级回退到文件系统时间。
7. 将命中的文件复制到目标目录，并保留时间元数据。
8. 对命中的 `HEIC`，如果存在对应的 Live Photo `MOV`，一并复制。

## 输出目录结构

输出根目录包含两个顶级分类：

```text
/output/
  only-two/
    2025-04/
  group/
    2025-04/
```

规则：
- `only-two/YYYY-MM/` 存放检测到的人脸总数正好为 2 且这两张脸匹配目标两人的图片。
- `group/YYYY-MM/` 存放检测到目标两人且检测到的人脸总数大于 2 的图片。
- 不保留源目录结构。
- 如果目标位置已经存在同名文件，工具会基于源路径或其他稳定标识追加确定性后缀，避免覆盖已有文件。

## Live Photo 配对规则

当命中的文件是 `HEIC` 时，工具必须在同一源目录中查找符合下述模式的隐藏 `MOV` 配对文件：

```text
.<HEIC filename>_<timestamp>.MOV
```

示例：
- 图片：`IMG_8175.HEIC`
- Live Photo 视频：`.IMG_8175_1771856408349261.MOV`

行为：
- `MOV` 文件永远不参与人脸识别。
- 如果找到了匹配的 `MOV`，则把它复制到与该 `HEIC` 相同的目标年月目录。
- 如果没找到匹配的 `MOV`，仍然复制该 `HEIC`，并在运行摘要中记录一条 warning。

## 架构

实现拆分为 5 个职责清晰的模块：

### 1. CLI

职责：
- 解析命令行参数
- 校验必填路径
- 协调整个端到端流程
- 输出摘要和 warning

### 2. 参考人脸加载器

职责：
- 加载每张参考照片
- 检测图片中的人脸
- 拒绝 0 张人脸或多张人脸的参考照片
- 为每个目标人物产出一个人脸编码

### 3. 递归扫描器

职责：
- 递归遍历输入目录
- 过滤受支持的图片类型
- 产出待匹配的候选图片文件
- 提供同目录 Live Photo `MOV` 配对查找能力

### 4. 匹配器

职责：
- 解码每张候选图片
- 检测人脸并提取编码
- 将候选图片中的人脸与两个目标编码分别比对
- 判断图片是否命中
- 将命中结果分类为 `only-two` 或 `group`

### 5. 导出器

职责：
- 从元数据解析目标年月目录
- 创建目标目录
- 在不覆盖无关已有文件的前提下复制文件
- 保留复制后文件的时间元数据

## 匹配规则

匹配逻辑：
- 只有当候选图片中至少有一张脸匹配参考人物 `A`，并且至少有一张脸匹配参考人物 `B` 时，这张图片才算命中。
- 候选图片中没有检测到人脸时，直接跳过。
- 候选图片解码失败时，跳过并记录 warning。

分类逻辑：
- 如果检测到的人脸总数正好是 `2`，分类为 `only-two`。
- 如果检测到的人脸总数大于 `2`，分类为 `group`。

第一版明确采用“检测到的人脸数量”来区分两类，而不是引入更复杂的场景语义判断。这样行为更明确，也更容易测试。

## 拍摄时间解析规则

年月目录命名基于照片拍摄时间。

解析顺序：
1. 图片元数据中的 EXIF 或媒体拍摄时间
2. 文件系统创建时间
3. 文件系统修改时间

格式：
- 目标子目录使用 `YYYY-MM`，例如 `2025-04`。

原因：
- 照片整理应该优先按拍摄时间归档，而不是按复制或导出时间归档。

## 元数据保留

复制文件后，工具必须尽量保留源文件的关键时间元数据。最低要求是通过标准复制方式保留修改时间和访问时间。在 macOS 上，如果可行，还应通过所选复制机制或额外步骤保留创建时间。

设计目标是让复制后的文件在 Finder 浏览和后续工具处理时，尽可能保留与原文件一致的时间表现。

## 错误处理

参考照片错误属于致命错误：
- 参考照片未检测到人脸：直接退出，并明确指出失败文件。
- 参考照片检测到多张人脸：直接退出，并明确指出失败文件。

候选图片错误属于非致命错误：
- 解码失败：跳过并记录 warning。
- 未检测到人脸：跳过，或者在摘要中计数。
- 命中的 `HEIC` 没有找到配对 `MOV`：仍复制 `HEIC`，并记录 warning。
- 拍摄时间元数据缺失：回退到文件系统时间。

CLI 摘要至少应报告：
- 扫描的文件数量
- `only-two` 中的命中数量
- `group` 中的命中数量
- 因解码或元数据问题跳过的文件数量
- 命中的 `HEIC` 中缺失配对 `MOV` 的数量

## 测试策略

第一版至少应包含以下边界测试：

1. 参考照片校验
- 0 张脸的参考照片会失败
- 1 张脸的参考照片会成功
- 多张脸的参考照片会失败

2. 递归扫描
- 包含受支持扩展名
- 忽略不受支持文件
- 能扫描嵌套子目录

3. Live Photo 配对查找
- `HEIC` 能找到匹配的隐藏 `MOV`
- 不匹配的隐藏 `MOV` 会被忽略

4. 命中分类
- 目标两人都命中且总人脸数为 `2` 时进入 `only-two`
- 目标两人都命中且总人脸数大于 `2` 时进入 `group`
- 缺少任一目标人物时不命中

5. 拍摄时间解析
- 有 EXIF 时间时优先使用 EXIF
- 没有元数据时回退到文件系统创建时间
- 创建时间不可用时最终回退到修改时间

6. 导出行为
- 目标路径使用 `YYYY-MM`
- 复制后保留时间元数据
- 同名冲突时不会覆盖已有文件
- 命中的 `HEIC` 会触发配对 `MOV` 的复制

## 风险与约束

- `face_recognition` 依赖 `dlib`，在部分机器上安装可能会比较麻烦，因此实现时需要把安装说明写清楚。
- 人脸匹配准确度会受到图片质量、姿态、光线以及底层模型偏差的影响。
- 分类规则依赖检测到的人脸数量，因此漏检或误检会影响图片是进入 `only-two` 还是 `group`。
- 保留 macOS 创建时间可能需要超出标准 Python 复制工具的额外处理。

## 成功标准

第一版满足以下条件即视为成功：
- 用户可以在 macOS 本地通过一条 CLI 命令运行工具。
- 工具可以递归扫描指定照片目录。
- 工具可以基于两张参考照片找出同时包含目标两人的图片。
- 工具可以把“仅两人合照”和“多人合照”分开输出。
- 工具可以把命中的文件复制到 `only-two/YYYY-MM` 和 `group/YYYY-MM`。
- 工具可以为命中的 `HEIC` 复制配对的 Live Photo `MOV` 文件。
- 复制后的文件在 Finder 和后续工具中能尽量保留与源文件一致的时间元数据表现。
